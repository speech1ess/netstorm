# Система логирования и отчётности PMI

Документ описывает, как в PMI организованы логи, артефакты тестов, управление конфигурациями и генерация HTML-отчётности.

## 1. Базовые сущности и идентификаторы

### 1.1. SESSION_ID (PMI_RUN_ID)
* **Переменная окружения:** `PMI_RUN_ID`
* **Формат:** `YYYYMMDD_HHMMSS` (например, `20260210_194850`)
* **Инициализация:** Задаётся в `ScenarioRunner.run_scenario()` при каждом новом запуске теста или серии.
* Определяет базовые каталоги для сессии:
    * `logs/<SESSION_ID>/` — сюда пишутся логи в реальном времени.
    * `results/<SESSION_ID>/` — сюда складываются итоговые отчёты и скопированные артефакты.

### 1.2. Акторы и Имена файлов (Actor Naming)
Внутри сессии каждый запускаемый инструмент (JMETER или TREX) получает уникальный идентификатор, который становится именем его лог-файла и JTL-отчета.
* **Формат:** `<tool>_<profile>_run<index>_<HHMMSS>`
* **Примеры:**
    * `jmeter_baseline_194850.log`
    * `trex_udp_flood_64b_194852.log`
* Это решает проблему параллельного запуска одинаковых инструментов — файлы никогда не перезапишут друг друга благодаря уникальному таймстампу старта.

## 2. Управление конфигурациями и Пресеты

Система поддерживает динамическую смену программ тестирования (Data-Driven Testing).

* **Менеджер конфигов (`scenario_loader.py`):** Сканирует папку `/opt/pmi/config/` на наличие файлов `test_program*.yaml` и `pmi_*.yaml`.
* **Состояние:** Выбранный конфиг сохраняется в переменной окружения `PMI_CURRENT_CONFIG`.
* **Пресеты (Presets):** Сценарии в YAML могут содержать блок `presets` (`low`, `medium`, `high`), которые на лету переопределяют `duration`, `override_tput`, `mult`, `threads` и `jprops` акторов, не требуя дублирования кода сценария.

## 3. Структура каталогов и артефакты

### 3.1. Логи (в процессе теста)
Все логи изначально пишутся в `logs/<SESSION_ID>/`:
* `pmi_session.log` — главный хронологический лог оркестратора.
* `jmeter_*.log` — индивидуальные логи каждого потока JMeter.
* `trex_*.log` — индивидуальные логи консоли TRex.
* `*.jtl` / `*_report/` — сырые метрики и дашборды JMeter (создаются прямо в папке `results`).
* `target_metrics.csv` — метрики здоровья цели (CPU/RAM), собираемые sidecar-монитором (только в режиме Backend).

### 3.2. Результаты (после теста)
По завершении сценария скрипт `generate_run_summary.py` **копирует** все использованные логи из `logs` в папку `results/<SESSION_ID>/`, чтобы отчёт был полностью автономным:
* `report_<SESSION_ID>.html` — финальный сводный отчёт.
* Все артефакты (JTL, логи, HTML-отчеты JMeter) копируются в эту же папку.

## 4. Оркестратор и обёртки

### 4.1. `ScenarioRunner` (`scenario_runner.py`)
Основные задачи:
1.  **Горячая загрузка:** Читает активный конфиг через `reload_config()`.
2.  **Применение пресетов:** Метод `_apply_preset_overrides()` накладывает параметры нагрузки поверх дефолтного сценария.
3.  **Оркестрация:** Генерирует `SESSION_ID`, собирает команды запуска (с пробросом `extra_args`, `jprops`, `payload`) и асинхронно спаунит процессы.
4.  **Batch Execution:** Метод `run_all_scenarios_batch()` позволяет запустить все сценарии из активного конфига друг за другом с заданным кулдауном.
5.  **Триггер отчёта:** По завершении вызывает `generate_run_summary.py` в фоновом режиме.

### 4.2. `jmeter_runner.py` / `trex_runner.py`
Выполняют роль адаптеров. Они принимают параметры от `ScenarioRunner` (tput, threads, duration), находят нужные файлы `.jmx` или `.py` в папке профилей и формируют финальную bash-команду (включая работу в `netns`, если это настроено).

## 5. Генерация отчётов (`generate_run_summary.py`)

Архитектура отчёта кардинально улучшена для точности данных:
1.  **Парсинг индивидуальных логов (`get_actor_stats_from_log`):** Вместо парсинга "сборной солянки" в `pmi_session.log`, скрипт читает индивидуальный скопированный файл инструмента (например, `jmeter_baseline_194850.log`) и вытаскивает финальную строку `summary = ...` (для JMeter) или финальный PPS (для TRex).
2.  **Построение HTML:**
    * **KPI Баннеры:** Выводят средний RPS, длительность и количество тестов (`Total Tests`).
    * **Execution Overview:** Каждому актору (инструменту) выделяется **отдельная строка** в таблице с индивидуальными показателями RPS и % ошибок. Статус `FAIL` ставится, если ошибки JMeter > 0.1%.
    * **Artifacts & Downloads:** Сгруппированные кнопки для открытия JTL, HTML-дашбордов и сырых логов.
    * **JSON Parsing:** Скрипт ищет в логах маркер `PMI_JSON_STATS={"key": "val"}` для извлечения точных метрик, игнорируя мусор в stdout.
    * **Target Health Chart:** График потребления ресурсов (CPU/RAM) целевого сервера, наложенный на временную шкалу теста. Строится на основе `target_metrics.csv` с использованием Chart.js.
    * **Log Viewer:** Основной лог `pmi_session.log` очищается от ANSI-цветов и встраивается прямо в веб-страницу.

## 6. Как этим пользоваться оператору (Workflows)

**Режим 1: Одиночный запуск с пресетом**
1. Запустить `./pmi_start.sh`.
2. В меню перейти в **Select Test Program File** и выбрать нужный YAML конфиг (например, `test_program2.yaml`).
3. Зайти в **Execute Scenarios** -> Выбрать группу (например, *Infrastructure Attacks*).
4. Выбрать сценарий `U1` -> Откроется подменю пресетов.
5. Выбрать `Medium Load`. Дождаться завершения теста.

**Режим 2: Пакетный запуск (Regression/Night Run)**
1. Запустить оркестратор, выбрать нужный конфиг.
2. В меню "Execute Scenarios" нажать **" RUN ALL SCENARIOS (Batch Execution)"**.
3. Указать глобальный пресет (например, `2` для Low Load) и интервал охлаждения (например, `60` секунд).
4. Оставить сервер работать. Система сама переберет все тесты, создаст для каждого отдельную сессию, отчёт и ZIP-архив.

**Доступ к результатам:**
* Глобальный индекс (календарь всех запусков): `http://<orchestrator-host>:8080/results/index.html`
* Все файлы сессии лежат по пути: `/opt/pmi/results/<SESSION_ID>/`. Отчёт можно сохранить как PDF прямо из браузера (стили адаптированы под печать).

## 7. Мониторинг Здоровья Цели (Sidecar Monitor)

Для корреляции атаки с состоянием сервера используется механизм **Sidecar Monitoring**:
1. При старте сценария (если включен режим `BACKEND`) запускается фоновый процесс `target_monitor.py`.
2. Раз в секунду он опрашивает endpoint `/` целевого сервера внутри netns.
3. Сервер (Python-воркер) возвращает JSON с метриками в теле ответа (маркер `PMI_JSON_STATS`).
4. Данные сохраняются в `logs/<SESSION_ID>/target_metrics.csv` в формате `time_offset,cpu,ram`.
5. При генерации отчета этот CSV превращается в интерактивный график.
