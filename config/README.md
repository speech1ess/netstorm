# 📝 Архитектура Конфигурации (NetStorm Orchestrator v1.0)

Этот раздел описывает Data-Driven устройство системы конфигурации NetStorm Orchestrator. Архитектура построена на принципе разделения слоев ответственности (от переменных окружения ОС до графического интерфейса), что позволяет гибко управлять оркестратором и стендом НТ без необходимости изменения исходного кода Python.

## 1. Слои Конфигурации (The Stack)

Система конфигурации состоит из четырех взаимосвязанных уровней. Все файлы расположены в директории `/opt/pmi/config/` (кроме `/opt/pmi/.env`).

| Слой | Файл | Назначение |
| :--- | :--- | :--- |
| **0. Environment** | `.env` | Переменные ОС. Определяет базовые пути и порты до старта Python. |
| **1. Infrastructure** | `global.yaml` | Топология и компоненты стенда. Сети (CIDR), VLAN, шлюзы, узлы, пути к бинарникам. |
| **2. Test Logic** | `test_program.yaml` | Программа и методика испытаний (ПМИ): DUT, логика тестов. сценарии, профили и маршруты атак, множители, пресеты. |
| **3. Interface** | `menu_structure.yaml` | UI/UX. Структура консольного меню и маппинг кнопок на Python-функции. |

> 💡 **Архитектурный принцип:** Data-driven. Скрипты, в идеале, не содержат хардкода и адаптируются под изменения данных в конфигурационных файлах. Интерфейс строится динамически из конфигурации, логика берется из программ, а IP-адреса — из глобальной топологии.

## 2. Разбор Слоев

### Слой 0: Environment (`.env`)
Базовый контракт между операционной системой (bash, systemd) и Python-оркестратором.
* `PMI_HOME`: Корневая директория проекта.
* `PMI_WEB_PORT`: Порт для FastAPI бэкенда.
* `PMI_CURRENT_CONFIG`: Указатель на активный файл программы испытаний (State).

### Слой 1: Infrastructure (`global.yaml`)
Единый источник правды о топологии и стенде.
* **`networks`**: Определяет пулы адресов (например, `trex_clients: "15.0.0.0/13"`).
* **`nodes`**: Описывает узлы (Target, TRex, JMeter) и их сервисы.
* **`paths`**: Стандартизированные пути ко всем компонентам системы.

### Слой 2: Logic (`test_program.yaml`)
Файл ПМИ, использующий возможности динамической шаблонизации Jinja2.
* **Шаблоны (Anchors `&`)**: Переиспользуемые блоки (например, маршруты атак).
* **Профили**: Связь логического имени (например, `syn_flood`) с физическим скриптом (`stl_syn_flood.py`) и базовыми переменными (`tunables`). Здесь применяется Jinja2 (`{{ ... }}`) для инъекции данных из `global.yaml`.
* **Сценарии**: Оркестрация акторов (инструментов) во времени, с поддержкой пресетов нагрузки (Low/Medium/High).

### Слой 3: Interface (`menu_structure.yaml`)
Определяет, как конечный пользователь взаимодействует с системой. Каждая кнопка меню (в TUI или Web) декларативно описывает свое поведение.

**Типы действий (Actions):**
* `type: "command"` — Выполнить bash-команду напрямую (например, проверка dmesg).
* `type: "python"` — Вызвать функцию из модуля (например, очистка логов).
* `type: "generator"` — Динамически построить подменю (например, список доступных сценариев из `test_program.yaml`).
* `type: "submenu"` — Статическая папка для группировки.

## 3. Пайплайн передачи данных (Data Flow)

Передача сетевых настроек от ПМИ (`test_program_*.yaml`) до генератора пакетов:

1. **Jinja2 Rendering (`ConfigLoader`):** При загрузке `test_program.yaml`, загрузчик парсит строки вида `{{ networks.external.subnets.trex_clients }}` и заменяет их на реальные CIDR-строки из кэша.
2. **Сетевой калькулятор и оркестратор сценариев (`scenario_runner.py`):** Оркестратор извлекает блок `tunables`. Если параметр `src_pool` задан в формате CIDR (`15.0.0.0/13`), он автоматически высчитывает первый и последний адреса подсети, формируя переменные `src_ip_start` и `src_ip_end`. Подготовленные переменные сериализуются в JSON-строку и передаются в виде аргумента.
3. **JSON Injection (`trex_runner.py`):** Раннер `trex_runner.py` получает дополнительные аргументы из JSON-строки, распаковывает их и передаёт напрямую в TRex API: `STLProfile.load_py(..., tunables=tunables)`.

---

## 4. Инструкция: Добавление нового сценария

Благодаря динамическим генераторам меню и Data-Driven подходу, добавление нового сценария не требует навыков программирования:

1. **Напишите профиль:** Создайте `.py` скрипт для TRex, принимающий динамические аргументы через `kwargs.get("dst_ip")`.
2. **Зарегистрируйте в `test_program.yaml`:** Добавьте скрипт в секцию `profiles` и передайте ему `tunables`.
3. **Создайте Сценарий:** Опишите вызов профиля в секции `scenarios`.
4. **Готово!** При запуске интерфейса, генератор меню автоматически прочитает файл конфигурации и отрисует кнопку для вашего нового сценария в разделе `Test & Scenarios Execution`.
