# Библиотека ядра оркестратора (/opt/pmi/lib/)

В данной директории содержатся основные Python-модули, реализующие всю логику работы PMI Load Test Orchestrator v3.0. 

Код разбит на независимые компоненты: от отрисовки интерактивного TUI-меню до низкоуровневого управления генераторами (TRex, JMeter) и парсинга логов.

---

## Структура модулей

### 1. Ядро и конфигурация (Core)
- `shared.py` — общие классы и утилиты. Содержит `ConfigLoader` (загрузчик YAML-конфигов с поддержкой шаблонизатора Jinja2), цвета консоли и обработчик прерываний (Trap Manager) для корректного завершения дочерних процессов.
- `pmi_logger.py` — централизованная система логирования (Singleton). Выводит цветные логи в консоль и пишет полные сессионные дампы в `pmi_session.log`.
- `config_manager.py` — логика TUI-раздела "Configuration Manager" (просмотр с подсветкой синтаксиса и редактирование `global.yaml` / `test_program.yaml`).

### 2. Оркестрация и выполнение тестов (Execution)
- `scenario_loader.py` — динамический парсер `test_program.yaml`. Генерирует структуру TUI-меню "PMI Scenarios" на лету.
- `scenario_runner.py` — главный движок тестов. Разбирает сценарий, применяет пресеты (low/medium/high), высчитывает тайминги и запускает нужные профили. Поддерживает запуск всех сценариев списком (Batch Execution).
- `trex_runner.py` — обертка над Stateless API TRex. Инициализирует L3 режим, делает ARP-resolve (MAC-адресов шлюза), отправляет трафик с нужного порта и асинхронно пушит метрики (TX/RX pps/bps) в VictoriaMetrics.
- `jmeter_runner.py` — обертка над JMeter CLI. Передает параметры (потоки, RPS, длительность), управляет сетевыми пространствами имен (netns) L7-генератора и генерирует отчеты (JTL, HTML).
- `profile_explorer.py` — логика "песочницы" (Profile Explorer). Позволяет просматривать директорию `/profiles/`, редактировать файлы и делать тестовые запуски (dry-run).

### 3. Пользовательский интерфейс (TUI)
- `menu_builder.py` — движок консольного меню. Парсит `menu_structure.yaml` и маршрутизирует выбор пользователя (запуск python-функций или bash-скриптов).

### 4. Мониторинг и управление целью (Target Management)
- `target_manager.py` — логика переключения NGINX между режимами отдачи статики (`STATIC`) и проксирования на Python (`BACKEND`).
- `system_monitor.py` — Python-воркер (веб-сервер), работающий на стороне цели (Target). Собирает метрики OS (CPU, RAM, Net).
- `target_monitor.py` — сборщик метрик. Опрашивает `system_monitor.py` во время теста и сохраняет данные в CSV.
- `system_status.py` — отрисовка виджета состояния системы в главном меню (CPU, RAM, статусы systemd, netns, trace).
- `diagnostics.py` — движок проверки здоровья стенда (проверка бинарников, синтаксиса YAML, скриптов, портов ZMQ, ICMP пинги).

### 5. Отчеты и аналитика (Reporting)
- `generate_run_summary.py` — генератор HTML-отчета для конкретной сессии. Парсит `pmi_session.log`, высчитывает средний RPS, долю ошибок, % пробития (leakage) для TRex-профилей.
- `generate_index.py` — генератор главного дашборда (календаря) `index.html`. Сканирует папку `results/`.
- `chart_builder.py` — конвертирует `target_metrics.csv` в красивые графики Chart.js для отчета.
- `html_templates.py` — хранилище HTML и CSS шаблонов для системы отчетов.

---

## Разработка

1. **Точка входа**: Оркестратор запускается через вызов `menu_builder.py`.
2. **Single Source of Truth**: Все пути, IP-адреса и порты **должны** читаться через `SharedConfig.get(...)`. Хардкод в этих файлах не допускается.
3. **Дочерние процессы**: Любой скрипт, запускающий внешние бинарники (например, `jmeter_runner`), должен регистрировать функцию очистки через `SharedTrap.register(cleanup_func)`, чтобы при нажатии `Ctrl+C` в главном меню все дочерние процессы (TRex, Java) убивались корректно.
